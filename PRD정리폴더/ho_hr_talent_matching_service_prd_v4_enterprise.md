# HO 인사실장님 인재개발·포스트매칭 서비스 구현 인터뷰 PRD

# 0. 문서 목적

이 문서는 HO 인사실장님이 구상하고 일부 구현해둔 **인재개발·포스트매칭 인사시스템**을 실제 서비스로 구현하기 위해 필요한 내용을 40분~1시간 인터뷰로 파악하고, 그 결과를 바로 개발 가능한 PRD로 전환하기 위한 문서다.

이 문서는 단순 질문지가 아니라 아래 3가지를 포함한다.

1. HO 인사실장님 인터뷰 진행 도구
2. 현재 구현된 프로토타입 분석 기준
3. 서비스 구현용 PRD 생성 기준

# 0-1. 추천 명칭

## 공식 문서명

**HO 인재개발·포스트매칭 서비스 구현 인터뷰 PRD**

## 내부 실행명

**인사실장님 인재판별기 서비스화 인터뷰 키트**

## 서비스 후보명

아래 중 하나를 임시명으로 사용한다.

1. 인재판별기
2. 포스트핏 매칭 시스템
3. Talent Fit Navigator
4. HR Talent Matching Platform
5. 사내 인재개발·발령추천 시스템

최종 서비스명은 인터뷰 후 확정한다.

# 0-2. 현재 전제

1. HO 인사실장님은 본인이 원하는 인사시스템/서비스를 일부 직접 구현해둔 상태다.
2. 향후 HTML 파일로 현재 구현물을 첨부하면, AI는 현재 반영현황을 분석하고 PRD에 반영해야 한다.
3. 단, 기존 구현을 무조건 보존하지 않는다.
4. 기존 구현물은 분석 후 **보존 / 수정 / 폐기 / 재구현** 대상으로 나누어 실장님에게 확인해야 한다.
5. 서비스의 핵심 방향은 사내 인재 개발관리다.
6. 회사 내 어떤 포스트/직무/직위가 비어있는지 확인해야 한다.
7. 비어있는 포스트에 적합한 인재를 자동 추천/제안해야 한다.
8. 추천된 인재를 실제 발령 검토까지 연결하는 것이 목표다.
9. 세부 기능으로는 직원 데이터를 기반으로 역량을 정량 분석하는 **인재판별기**가 필요하다.
10. 인재판별기 로직은 일부 구현되어 있을 가능성이 있다.
11. 인터뷰 시간은 40분~1시간 이내로 제한한다.

# 0-3. HTML 첨부 시 분석 원칙

실장님이 구현한 서비스 HTML 파일이 첨부되면, AI는 먼저 아래 항목을 분석한다.

1. 프로젝트 기술스택
2. 폴더/파일 구조
3. 현재 구현된 화면
4. 현재 구현된 기능
5. 현재 구현된 데이터 구조
6. 현재 구현된 인재판별 로직
7. 현재 구현된 추천/점수 산식
8. 미구현 기능
9. 오류 가능성이 있는 코드
10. 서비스화 시 보완해야 할 구조
11. 보안상 위험한 부분
12. 개인정보/민감정보 처리 위험

# 0-4. 기존 구현물 반영 여부 확인 원칙

기존 구현물은 자동으로 유지하지 않는다.

HTML 분석 후 AI는 반드시 아래 표를 작성하고 실장님에게 확인한다.

| 구분 | 항목 | 현재 상태 | AI 1차 판단 | 실장님 확인 | 최종 처리 |
|---|---|---|---|---|---|
| 화면 |  | 구현/미구현/오류 | 보존/수정/폐기/재구현 |  |  |
| 기능 |  | 구현/미구현/오류 | 보존/수정/폐기/재구현 |  |  |
| 로직 |  | 구현/미구현/오류 | 보존/수정/폐기/재구현 |  |  |
| 데이터 |  | 구현/미구현/오류 | 보존/수정/폐기/재구현 |  |  |
| UI |  | 구현/미구현/오류 | 보존/수정/폐기/재구현 |  |  |

처리 기준은 아래와 같다.

1. **보존**: 그대로 유지하고 PRD에 반영한다.
2. **수정**: 방향은 유지하되 구조, UI, 코드, 데이터 모델을 개선한다.
3. **폐기**: 서비스 목표와 맞지 않거나 품질/보안상 위험해 사용하지 않는다.
4. **재구현**: 아이디어는 참고하되 새 구조로 다시 만든다.

AI는 아래 질문을 반드시 포함한다.

1. 현재 구현한 기능 중 반드시 살리고 싶은 기능은 무엇인가요?
2. 현재 구현했지만 버려도 되는 기능은 무엇인가요?
3. 아이디어는 좋지만 새로 다시 만드는 게 나은 기능은 무엇인가요?
4. 현재 화면 중 제안서나 시연에 활용하고 싶은 화면은 무엇인가요?
5. 현재 로직 중 실제 서비스의 핵심으로 유지하고 싶은 산식/기준은 무엇인가요?
6. 현재 구현에서 보안상 또는 개인정보상 걱정되는 부분은 무엇인가요?


# 0-5. HTML 파일 분석 기준

기존 구현물이 HTML 파일로 제공되는 경우, AI는 아래 항목을 우선 분석한다.

1. HTML 전체 구조
2. 화면 섹션 구성
3. 입력 폼 항목
4. 버튼/이벤트 흐름
5. JavaScript 로직
6. CSS 스타일 구조
7. 인재판별기 관련 계산식
8. 후보 추천 관련 조건문
9. 샘플 데이터 구조
10. 브라우저에서 실제로 동작 가능한 기능
11. 현재 화면에서 보이는 기능과 코드상 존재하지만 화면에 드러나지 않는 기능
12. 서비스화 시 Next.js/TypeScript/Supabase 구조로 옮겨야 할 기능
13. 유지할 UI 요소
14. 버릴 UI 요소
15. 새로 설계해야 할 화면/기능

HTML 파일 분석 후에도 AI는 기존 구현을 무조건 보존하지 않는다.

반드시 아래 4가지로 나누어 실장님에게 확인한다.

1. 보존
2. 수정
3. 폐기
4. 재구현


# 0-6. 추가 확정 요구사항

아래 항목은 사용자가 추가 확정한 요구사항이므로, 인터뷰와 최종 PRD에 반드시 반영한다.

## 1. 기업용 UI/UX 톤

이 서비스는 기업용 HR 서비스이므로 이모지, 과한 색상, 과도한 애니메이션, 화려한 장식 요소를 사용하지 않는다.

UI 방향은 아래 기준을 따른다.

1. 담백하고 직관적인 기업용 SaaS UI
2. 토스 UI처럼 간결하고 정보 위계가 명확한 화면
3. Workday UI처럼 HR 업무에 적합한 안정적이고 신뢰감 있는 화면
4. 흰색/회색 기반의 차분한 배경
5. 포인트 컬러는 1~2개만 사용
6. 상태값 색상은 최소화
7. 이모지는 원칙적으로 사용하지 않음
8. 표, 카드, 필터, 상세 패널 중심의 업무형 UI
9. 경영진/인사실장/실무자가 모두 이해할 수 있는 직관적 레이아웃
10. 제안서에 캡처해도 가볍거나 장난스러워 보이지 않는 디자인

인터뷰 시 반드시 아래 질문을 포함한다.

1. 서비스 UI는 토스처럼 간결한 스타일과 Workday처럼 정통 HR SaaS 스타일 중 어디에 더 가까워야 하나요?
2. 회사 내부 경영진에게 보여줄 때 피해야 할 색상, 표현, 화면 분위기가 있나요?
3. 제안서 캡처용 화면에서 반드시 전문적으로 보여야 하는 화면은 무엇인가요?

## 2. 단계별 구현 범위 확인

PRD는 최종적으로 모든 기능을 구현할 수 있을 만큼 상세하게 작성한다.

단, 실제 개발은 한 번에 모든 기능을 만들지 않고 단계별로 나누어 구현한다.

인터뷰 중 반드시 아래 내용을 확인한다.

1. 1차 MVP에서 반드시 구현할 기능
2. 2차에서 구현할 기능
3. 3차 이후 고도화 기능
4. 지금은 PRD에만 남기고 구현하지 않을 기능
5. 이번 개발에서 어디까지 구현할지
6. 내부 시연용 범위
7. 고객 테스트용 범위
8. 외부 제안용 데모 범위
9. 정식 서비스화 범위

구현 단계는 기본적으로 아래처럼 나눈다.

### Phase 1. 인터뷰/시연용 최소 프로토타입

목적:
- 핵심 서비스 흐름을 보여준다.
- 포스트 선택 → 후보 추천 → 추천 사유 확인 흐름을 구현한다.

포함 후보:
1. 샘플 데이터 기반 대시보드
2. 포스트 목록
3. 후보자 목록
4. 인재판별기 기본 점수 계산
5. 추천 후보 표시
6. 추천 사유 표시
7. 기본 피드백 수집

### Phase 2. 고객 테스트용 MVP

목적:
- 실제 고객이 제한된 시나리오를 수행하고 리뷰를 남길 수 있게 한다.

포함 후보:
1. 고객 테스트 모드
2. 사용 리뷰 입력
3. 후보 비교
4. 발령 검토 상태 관리
5. 관리자용 데이터 수정
6. 샘플 데이터 관리
7. Supabase 저장

### Phase 3. 내부 운영형 서비스

목적:
- 인사팀/협업부서가 실제 운영 흐름을 관리할 수 있게 한다.

포함 후보:
1. 권한별 화면
2. 협업부서 입력/검토 화면
3. 조직별 포스트 관리
4. 리뷰/승인 히스토리
5. 보안/접근 로그
6. 데이터 업로드/관리

### Phase 4. 외부 B2B 판매형 서비스

목적:
- 외부 회사에 판매 가능한 형태로 기능과 UI를 고도화한다.

포함 후보:
1. 회사별 테넌트 구조
2. 고객사별 설정
3. 가격/플랜 관리
4. 관리자 콘솔
5. 온보딩 화면
6. 사용량/리포트
7. 제안서/보고서 자동 생성

## 3. 샘플 데이터 및 데이터 구조 확인

인사서비스는 데이터 구조가 핵심이므로, 인터뷰 중 샘플 데이터와 데이터 구조를 반드시 확인한다.

확인해야 할 데이터는 아래와 같다.

1. 포스트 샘플 데이터
2. 후보자/직원 샘플 데이터
3. 역량 항목 샘플
4. 성과/평가 데이터 샘플
5. 직무경험 데이터 샘플
6. 조직/부서/직위 데이터 샘플
7. 발령 검토 상태 샘플
8. 인재판별기 점수 산식 샘플
9. 추천 결과 샘플
10. 고객 테스트 리뷰 샘플

인터뷰 시 반드시 아래 질문을 포함한다.

1. MVP에서 사용할 샘플 직원 데이터는 몇 명 정도가 적절한가요?
2. 포스트 샘플은 몇 개 정도 필요할까요?
3. 실제 직원명 대신 익명/가명 데이터를 사용해야 하나요?
4. 후보자 추천에 필요한 최소 데이터 컬럼은 무엇인가요?
5. 인재판별기 점수 계산에 반드시 들어가야 하는 데이터는 무엇인가요?
6. 현재 HTML 안에 포함된 샘플 데이터가 있다면 그대로 쓸까요, 새로 만들까요?
7. 데이터 구조는 실제 HR 데이터 구조와 비슷해야 하나요, 시연용으로 단순화해도 되나요?

## 4. 보안/개인정보 체크

이 서비스는 인사정보, 평가정보, 역량정보, 발령 검토 정보를 다루므로 보안과 개인정보 검토가 필수다.

단, 1차 사용 환경은 사내망이므로 당장 별도의 보안 인프라를 추가 구축하는 것을 전제로 하지 않는다.

MVP 보안 원칙은 아래와 같다.

1. 우선 사내망 사용을 전제로 한다.
2. 외부 공개 URL에는 실제 인사정보를 넣지 않는다.
3. 외부 시연/고객 테스트 시에는 샘플 또는 익명 데이터를 사용한다.
4. 실제 직원명, 사번, 평가등급, 민감 메모는 제한적으로 사용한다.
5. 사용자 권한은 최소한의 역할 구분부터 설계한다.
6. HR 관리자와 일반 조회자의 권한을 구분한다.
7. 추천 결과는 최종 인사결정이 아니라 참고/검토용으로 표시한다.
8. 접속 로그, 수정 이력, 추천 결과 생성 이력은 향후 확장 항목으로 남긴다.
9. 개인정보/평가정보를 외부 고객에게 노출하지 않는다.
10. 보안 인프라 추가 구축은 Phase 3 또는 Phase 4에서 검토한다.

인터뷰 시 반드시 아래 질문을 포함한다.

1. 1차 버전은 사내망에서만 사용할 예정인가요?
2. 실제 직원 데이터를 사용할까요, 샘플/익명 데이터를 사용할까요?
3. 평가정보, 성과정보, 리더십 정보 중 외부 시연에서 제외해야 할 항목은 무엇인가요?
4. 권한은 인사실장, 인사팀, 협업부서, 조회자 중 어떻게 나눠야 하나요?
5. 외부 고객 테스트 시 어떤 데이터까지 보여줘도 되나요?
6. 보안상 절대 화면에 노출되면 안 되는 항목은 무엇인가요?

## 5. 협업부서 및 공동 사용자 확인

이 서비스는 HO 인사실장만 쓰는 도구가 아닐 수 있다.

협업부서 또는 공동 사용자가 있다면, 해당 사용자별 화면과 기능을 함께 설계해야 한다.

확인할 사용자 그룹은 아래와 같다.

1. HO 인사실장
2. 인사팀 실무자
3. HRBP
4. 계열사 대표
5. 계열사 인사실장
6. 사업부장
7. 현업 부서장
8. 후보자 평가/검토 참여자
9. 외부 고객 테스트 참여자
10. 시스템 관리자

인터뷰 시 반드시 아래 질문을 포함한다.

1. 이 서비스를 실장님 외에 함께 사용할 사람은 누구인가요?
2. 협업부서가 데이터를 입력해야 하나요, 조회만 하면 되나요?
3. 사업부장이나 대표는 어떤 화면만 보면 되나요?
4. HRBP나 인사팀 실무자는 어떤 기능이 필요하나요?
5. 후보자 추천 결과에 대해 의견을 남기는 사람이 있나요?
6. 외부 고객 테스트 참여자는 어떤 제한된 화면을 봐야 하나요?
7. 사용자별로 보여주면 안 되는 정보는 무엇인가요?

기본 권한 구조는 아래처럼 설계한다.

| 역할 | 주요 권한 |
|---|---|
| Super Admin | 전체 설정, 데이터 관리, 권한 관리 |
| HR Admin | 포스트/후보자/추천로직 관리 |
| HR Reviewer | 추천 결과 검토, 메모, 상태 변경 |
| Executive Viewer | 경영진 요약 화면 조회 |
| Department Reviewer | 본인 조직 관련 후보/포스트 제한 조회 및 의견 입력 |
| Demo User | 샘플 데이터 기반 제한 체험 |

# 1. 작업 목적

HO 인사실장님이 구상한 인재개발·포스트매칭 시스템을 실제 사용 가능한 프로토타입/MVP로 구현한다.

이 서비스는 단순 인사조회 시스템이 아니라, 아래 기능을 목표로 한다.

1. 회사 내 현재 비어 있는 포스트를 파악한다.
2. 각 포스트에 필요한 역량, 직무경험, 직위, 평가, 성과, 리더십 요건을 정의한다.
3. 직원들의 인사 데이터를 기반으로 적합도를 계산한다.
4. 특정 포스트에 적합한 후보자를 추천한다.
5. 추천 사유와 리스크를 설명한다.
6. 인사실장 또는 의사결정자가 후보를 검토한다.
7. 실제 발령 검토/추천/승인 프로세스로 연결한다.
8. 실제 고객에게 프로토타입을 사용하게 하고 리뷰를 받을 수 있도록 한다.

# 2. 현재 문제

현재 인재 배치와 포스트 관리 과정에는 아래 문제가 있을 가능성이 높다.

1. 어떤 포스트가 비어 있는지 한눈에 보기 어렵다.
2. 포스트별 필요 역량이 정량화되어 있지 않다.
3. 인재 추천이 경험과 기억에 의존한다.
4. 후보자의 강점과 리스크를 비교하기 어렵다.
5. 발령 후보 검토 과정이 문서, 엑셀, 메신저, 회의 중심으로 흩어져 있다.
6. 인재 육성, 승계, 발령, 직무경험 데이터가 연결되어 있지 않다.
7. 실장님이 일부 구현한 로직이 있어도 서비스 구조와 화면 흐름이 완성되어 있지 않을 수 있다.
8. 외부 고객 또는 계열사에 보여줄 수준의 프로토타입 정리가 필요하다.

# 3. 개선 목표

1. 포스트 현황을 한눈에 보여준다.
2. 포스트별 필요 요건을 표준화한다.
3. 직원별 인재 데이터와 역량 데이터를 구조화한다.
4. 인재판별기 로직으로 후보 적합도를 계산한다.
5. 포스트별 추천 후보자를 자동 제안한다.
6. 추천 사유, 부족 역량, 발령 리스크를 함께 보여준다.
7. 인사실장이 후보를 검토하고 상태를 관리할 수 있게 한다.
8. 고객 테스트를 위한 프로토타입 사용 시나리오를 제공한다.
9. B2B 제안서에 들어갈 서비스 최종 그림과 구현현황을 정리할 수 있게 한다.

# 6. 핵심 개념 정의

# 6-1. 포스트

포스트는 회사 내 특정 직무/직위/역할을 의미한다.

예시:

1. 점장
2. 영업팀장
3. MD팀장

5. 물류기획 담당

# 6-3. 후보자

특정 포스트에 추천될 수 있는 직원 또는 인재풀 구성원이다.

# 6-4. 인재판별기

직원 데이터를 기반으로 특정 포스트에 적합한지 점수화하고 추천하는 로직이다.

# 6-5. 적합도 점수

후보자가 특정 포스트에 얼마나 적합한지 나타내는 정량 점수다.

# 7. 40분~1시간 인터뷰 운영안

# 7-1. 운영 원칙

1. 질문은 전체 질문뱅크가 아니라 핵심 질문 중심으로 진행한다.
2. 한 번에 3개 이하로 질문한다.
3. HTML 파일이 첨부되어 있으면 현재 구현현황 분석을 먼저 한다.
4. 기존 구현물은 무조건 보존하지 않는다.
5. 기존 구현물은 보존/수정/폐기/재구현 대상을 분류한 뒤 실장님에게 확인한다.
6. 세부 로직을 모르면 “[추후확인]”으로 남긴다.
7. 인터뷰 종료 후 바로 구현 PRD로 변환 가능한 답변을 수집한다.

# 7-2. 시간 배분

| 시간 | 단계 | 목적 |
|---|---|---|
| 0~5분 | 오프닝 | 인터뷰 목적 확인 |
| 5~15분 | 현재 구현현황 | 이미 만든 서비스/로직 파악 |
| 15~20분 | 보존/폐기 판단 | 기존 구현 중 살릴 것과 버릴 것 확인 |
| 20~30분 | 서비스 큰 그림 | 최종 서비스 목적과 사용자 파악 |
| 30~45분 | 핵심 기능 | 포스트, 후보자, 인재판별기, 추천 로직 파악 |
| 45~55분 | 화면/데이터 | 필요한 화면과 데이터 구조 파악 |
| 55~60분 | 테스트/일정 | 고객 테스트, 구현 우선순위, 일정 확인 |

# 11. UI/UX 요구사항

1. 첫 화면은 경영진/인사실장이 바로 이해할 수 있는 요약 대시보드로 구성한다.
2. 포스트 중심으로 화면을 설계한다.
3. 포스트를 클릭하면 추천 후보가 바로 보여야 한다.
4. 후보 추천 결과는 점수, 추천 사유, 리스크가 함께 보여야 한다.
5. 후보 비교는 2~5명까지 가능하게 한다.
6. 점수 산식은 숨기지 말고 관리자 화면에서 확인할 수 있어야 한다.
7. 실제 인사결정으로 오해되지 않도록 “추천/검토용” 문구를 표시한다.
8. 고객 테스트용 모드는 더 단순한 화면으로 제공한다.
9. 개인정보/민감정보는 샘플 데이터 또는 익명 데이터로 테스트한다.
10. 제안서에 넣을 수 있는 화면 캡처가 가능하도록 UI를 정돈한다.
11. 기업용 서비스이므로 이모지, 과한 컬러, 장식적 요소를 사용하지 않는다.
12. 토스 UI처럼 간결하고 Workday UI처럼 안정적인 HR SaaS 톤을 따른다.

2. 파일이 있으면 새로 만들지 말고 기존 파일에 누적 작성한다.

6. DATA/DB/API 변경이 없으면 “변경 없음”이라고 명시한다.
