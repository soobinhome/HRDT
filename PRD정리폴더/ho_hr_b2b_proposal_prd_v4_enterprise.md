# HO 인사실장님 인재개발 서비스 B2B 제안서 생성 인터뷰 PRD

# 0. 문서 목적

이 문서는 HO 인사실장님이 구상하고 일부 구현한 **인재개발·포스트매칭 서비스**를 기반으로, 계열사 내부 법인과 외부 회사 경영진/인사실장을 대상으로 제안할 **B2B 제안서 PPT**를 생성하기 위한 인터뷰 PRD다.

최종 목적은 아래 3가지다.

1. 서비스의 최종 그림을 경영진이 이해하기 쉽게 정리한다.
2. 현재 구현현황과 개발 완료 타임라인을 제안서에 넣는다.
3. 예상 판매가, 목표 판매가, 과금모델을 정리해 B2B 판매 제안서로 만든다.

# 0-3. 현재 구현물 반영 원칙

제안서에는 현재 구현현황을 넣어야 하지만, 기존 구현물을 무조건 긍정적으로 포장하지 않는다.

HTML 파일 또는 현재 구현물이 제공되면 AI는 아래 기준으로 분석한다.

1. 제안서에 넣어도 되는 화면
2. 제안서에 넣기에는 부족한 화면
3. 시연 가능한 기능
4. 시연하면 안 되는 기능
5. 현재 로직 중 강조할 수 있는 부분
6. 현재 로직 중 보완 후 언급해야 하는 부분
7. 버려도 되는 기능
8. 새로 다시 만들어야 하는 기능
9. 개발 완료 타임라인에 반영해야 하는 미구현 기능

AI는 반드시 아래 질문을 한다.

1. 현재 구현된 화면 중 제안서에 캡처로 넣어도 되는 화면은 무엇인가요?
2. 현재 구현된 기능 중 시연 가능한 기능은 무엇인가요?
3. 아직 제안서에서 강조하면 안 되는 기능은 무엇인가요?
4. 현재 구현했지만 고객에게 보여주지 않아도 되는 기능은 무엇인가요?
5. 제안서에서는 현재 구현현황을 어느 정도까지 공개하고 싶나요?
6. 현재 구현물 대신 최종 완성 그림 중심으로 제안하고 싶은가요?

# 0-4. 제안서 핵심 메시지

제안서는 아래 메시지를 중심으로 작성한다.

1. 회사에는 늘 빈 포스트와 승계 리스크가 존재한다.
2. 좋은 인재는 있지만, 누가 어떤 포스트에 적합한지 정량적으로 판단하기 어렵다.
3. 이 서비스는 포스트 요건과 인재 데이터를 연결해 적합 후보를 추천한다.
4. 인사 의사결정을 자동화하는 것이 아니라, 더 빠르고 객관적인 검토를 돕는다.
5. 프로토타입은 일부 구현되어 있으나, 제안서에는 보존/수정/폐기/재구현 판단이 끝난 내용만 반영한다.
6. 내부 계열사 적용 후 외부 B2B 솔루션으로 확장할 수 있다.


# 0-5. HTML 파일 분석 기준

기존 구현물이 HTML 파일로 제공되는 경우, AI는 아래 항목을 우선 분석한다.

1. HTML 전체 구조
2. 화면 섹션 구성
3. 입력 폼 항목
4. 버튼/이벤트 흐름
5. JavaScript 로직
6. CSS 스타일 구조
7. 인재판별기 관련 계산식
8. 후보 추천 관련 조건문
9. 샘플 데이터 구조
10. 브라우저에서 실제로 동작 가능한 기능
11. 현재 화면에서 보이는 기능과 코드상 존재하지만 화면에 드러나지 않는 기능
12. 서비스화 시 Next.js/TypeScript/Supabase 구조로 옮겨야 할 기능
13. 유지할 UI 요소
14. 버릴 UI 요소
15. 새로 설계해야 할 화면/기능

HTML 파일 분석 후에도 AI는 기존 구현을 무조건 보존하지 않는다.

반드시 아래 4가지로 나누어 실장님에게 확인한다.

1. 보존
2. 수정
3. 폐기
4. 재구현


# 0-6. 추가 확정 요구사항

아래 항목은 사용자가 추가 확정한 요구사항이므로, 인터뷰와 최종 PRD에 반드시 반영한다.

## 1. 기업용 UI/UX 톤

이 서비스는 기업용 HR 서비스이므로 이모지, 과한 색상, 과도한 애니메이션, 화려한 장식 요소를 사용하지 않는다.

UI 방향은 아래 기준을 따른다.

1. 담백하고 직관적인 기업용 SaaS UI
2. 토스 UI처럼 간결하고 정보 위계가 명확한 화면
3. Workday UI처럼 HR 업무에 적합한 안정적이고 신뢰감 있는 화면
4. 흰색/회색 기반의 차분한 배경
5. 포인트 컬러는 1~2개만 사용
6. 상태값 색상은 최소화
7. 이모지는 원칙적으로 사용하지 않음
8. 표, 카드, 필터, 상세 패널 중심의 업무형 UI
9. 경영진/인사실장/실무자가 모두 이해할 수 있는 직관적 레이아웃
10. 제안서에 캡처해도 가볍거나 장난스러워 보이지 않는 디자인

인터뷰 시 반드시 아래 질문을 포함한다.

1. 서비스 UI는 토스처럼 간결한 스타일과 Workday처럼 정통 HR SaaS 스타일 중 어디에 더 가까워야 하나요?
2. 회사 내부 경영진에게 보여줄 때 피해야 할 색상, 표현, 화면 분위기가 있나요?
3. 제안서 캡처용 화면에서 반드시 전문적으로 보여야 하는 화면은 무엇인가요?

## 2. 단계별 구현 범위 확인

PRD는 최종적으로 모든 기능을 구현할 수 있을 만큼 상세하게 작성한다.

단, 실제 개발은 한 번에 모든 기능을 만들지 않고 단계별로 나누어 구현한다.

인터뷰 중 반드시 아래 내용을 확인한다.

1. 1차 MVP에서 반드시 구현할 기능
2. 2차에서 구현할 기능
3. 3차 이후 고도화 기능
4. 지금은 PRD에만 남기고 구현하지 않을 기능
5. 이번 개발에서 어디까지 구현할지
6. 내부 시연용 범위
7. 고객 테스트용 범위
8. 외부 제안용 데모 범위
9. 정식 서비스화 범위

구현 단계는 기본적으로 아래처럼 나눈다.

### Phase 1. 인터뷰/시연용 최소 프로토타입

목적:
- 핵심 서비스 흐름을 보여준다.
- 포스트 선택 → 후보 추천 → 추천 사유 확인 흐름을 구현한다.

포함 후보:
1. 샘플 데이터 기반 대시보드
2. 포스트 목록
3. 후보자 목록
4. 인재판별기 기본 점수 계산
5. 추천 후보 표시
6. 추천 사유 표시
7. 기본 피드백 수집

### Phase 2. 고객 테스트용 MVP

목적:
- 실제 고객이 제한된 시나리오를 수행하고 리뷰를 남길 수 있게 한다.

포함 후보:
1. 고객 테스트 모드
2. 사용 리뷰 입력
3. 후보 비교
4. 발령 검토 상태 관리
5. 관리자용 데이터 수정
6. 샘플 데이터 관리
7. Supabase 저장

### Phase 3. 내부 운영형 서비스

목적:
- 인사팀/협업부서가 실제 운영 흐름을 관리할 수 있게 한다.

포함 후보:
1. 권한별 화면
2. 협업부서 입력/검토 화면
3. 조직별 포스트 관리
4. 리뷰/승인 히스토리
5. 보안/접근 로그
6. 데이터 업로드/관리

### Phase 4. 외부 B2B 판매형 서비스

목적:
- 외부 회사에 판매 가능한 형태로 기능과 UI를 고도화한다.

포함 후보:
1. 회사별 테넌트 구조
2. 고객사별 설정
3. 가격/플랜 관리
4. 관리자 콘솔
5. 온보딩 화면
6. 사용량/리포트
7. 제안서/보고서 자동 생성

## 3. 샘플 데이터 및 데이터 구조 확인

인사서비스는 데이터 구조가 핵심이므로, 인터뷰 중 샘플 데이터와 데이터 구조를 반드시 확인한다.

확인해야 할 데이터는 아래와 같다.

1. 포스트 샘플 데이터
2. 후보자/직원 샘플 데이터
3. 역량 항목 샘플
4. 성과/평가 데이터 샘플
5. 직무경험 데이터 샘플
6. 조직/부서/직위 데이터 샘플
7. 발령 검토 상태 샘플
8. 인재판별기 점수 산식 샘플
9. 추천 결과 샘플
10. 고객 테스트 리뷰 샘플

인터뷰 시 반드시 아래 질문을 포함한다.

1. MVP에서 사용할 샘플 직원 데이터는 몇 명 정도가 적절한가요?
2. 포스트 샘플은 몇 개 정도 필요할까요?
3. 실제 직원명 대신 익명/가명 데이터를 사용해야 하나요?
4. 후보자 추천에 필요한 최소 데이터 컬럼은 무엇인가요?
5. 인재판별기 점수 계산에 반드시 들어가야 하는 데이터는 무엇인가요?
6. 현재 HTML 안에 포함된 샘플 데이터가 있다면 그대로 쓸까요, 새로 만들까요?
7. 데이터 구조는 실제 HR 데이터 구조와 비슷해야 하나요, 시연용으로 단순화해도 되나요?

## 4. 보안/개인정보 체크

이 서비스는 인사정보, 평가정보, 역량정보, 발령 검토 정보를 다루므로 보안과 개인정보 검토가 필수다.

단, 1차 사용 환경은 사내망이므로 당장 별도의 보안 인프라를 추가 구축하는 것을 전제로 하지 않는다.

MVP 보안 원칙은 아래와 같다.

1. 우선 사내망 사용을 전제로 한다.
2. 외부 공개 URL에는 실제 인사정보를 넣지 않는다.
3. 외부 시연/고객 테스트 시에는 샘플 또는 익명 데이터를 사용한다.
4. 실제 직원명, 사번, 평가등급, 민감 메모는 제한적으로 사용한다.
5. 사용자 권한은 최소한의 역할 구분부터 설계한다.
6. HR 관리자와 일반 조회자의 권한을 구분한다.
7. 추천 결과는 최종 인사결정이 아니라 참고/검토용으로 표시한다.
8. 접속 로그, 수정 이력, 추천 결과 생성 이력은 향후 확장 항목으로 남긴다.
9. 개인정보/평가정보를 외부 고객에게 노출하지 않는다.
10. 보안 인프라 추가 구축은 Phase 3 또는 Phase 4에서 검토한다.

인터뷰 시 반드시 아래 질문을 포함한다.

1. 1차 버전은 사내망에서만 사용할 예정인가요?
2. 실제 직원 데이터를 사용할까요, 샘플/익명 데이터를 사용할까요?
3. 평가정보, 성과정보, 리더십 정보 중 외부 시연에서 제외해야 할 항목은 무엇인가요?
4. 권한은 인사실장, 인사팀, 협업부서, 조회자 중 어떻게 나눠야 하나요?
5. 외부 고객 테스트 시 어떤 데이터까지 보여줘도 되나요?
6. 보안상 절대 화면에 노출되면 안 되는 항목은 무엇인가요?

## 5. 협업부서 및 공동 사용자 확인

이 서비스는 HO 인사실장만 쓰는 도구가 아닐 수 있다.

협업부서 또는 공동 사용자가 있다면, 해당 사용자별 화면과 기능을 함께 설계해야 한다.

확인할 사용자 그룹은 아래와 같다.

1. HO 인사실장
2. 인사팀 실무자
3. HRBP
4. 계열사 대표
5. 계열사 인사실장
6. 사업부장
7. 현업 부서장
8. 후보자 평가/검토 참여자
9. 외부 고객 테스트 참여자
10. 시스템 관리자

인터뷰 시 반드시 아래 질문을 포함한다.

1. 이 서비스를 실장님 외에 함께 사용할 사람은 누구인가요?
2. 협업부서가 데이터를 입력해야 하나요, 조회만 하면 되나요?
3. 사업부장이나 대표는 어떤 화면만 보면 되나요?
4. HRBP나 인사팀 실무자는 어떤 기능이 필요하나요?
5. 후보자 추천 결과에 대해 의견을 남기는 사람이 있나요?
6. 외부 고객 테스트 참여자는 어떤 제한된 화면을 봐야 하나요?

# 4-2. 인터뷰 운영 원칙

1. 질문은 한 번에 3개 이하로 한다.
2. 답변이 모호하면 고객, 가격, 기능, 일정 기준으로 다시 묻는다.
3. 제안서에 쓸 표현으로 바로 바꿔가며 정리한다.
4. 실장님이 모르는 내용은 `[추후확인]`으로 남긴다.
5. 서비스 구현 PRD와 충돌하지 않도록 기능 범위와 개발 일정은 일치시킨다.
6. 기존 구현물은 무조건 제안서에 넣지 않는다.
7. 제안서 반영 기능과 제외 기능을 실장님에게 확인한다.

7. 도입한다면 어느 부서에서 먼저 사용할 것 같나요?
8. 구매 또는 PoC를 검토할 의향이 있나요?
9. 적정하다고 생각하는 월 사용료는 어느 정도인가요?
10. 제안서에서 더 강조해야 할 내용은 무엇인가요?

# 10. 디자인 톤

# 10-1. 경영진용 톤

1. 짧고 명확하게
2. 문제 → 해결 → 효과 → 비용 → 일정 순서
3. 숫자와 의사결정 포인트 중심
4. 복잡한 HR 용어 최소화
5. “인사결정 자동화”가 아니라 “의사결정 지원”으로 표현

# 10-2. 인사실장용 톤

1. 실무 적용성 강조
2. 포스트/후보자/발령 검토 흐름 강조
3. 평가데이터와 추천 로직의 투명성 강조
4. 보안과 개인정보 관리 강조
6. 기업용 SaaS에 맞는 담백하고 신뢰감 있는 디자인 강조
5. 고객 리뷰와 개선 사이클 강조

# 11. 주의사항

1. AI가 인사발령을 자동으로 결정한다고 표현하지 않는다.
2. 추천 결과는 참고용/검토용이라고 명확히 표현한다.
3. 개인정보와 평가정보를 다루므로 보안 검토가 필요하다고 명시한다.
4. 실제 고객 테스트 시 샘플 또는 익명 데이터를 사용한다고 명시한다.
4-1. 1차 사용 환경은 사내망이므로 당장 별도 보안 인프라 추가 구축이 필수는 아니지만, 데이터 노출 범위와 권한은 반드시 설계한다고 명시한다.
5. 판매가와 목표가는 확정 금액이 아니라 인터뷰 기반 가설로 구분한다.
6. 개발 완료 타임라인은 HTML 분석 후 조정될 수 있다고 명시한다.
7. 현재 구현된 기능이라고 해서 모두 제안서에 넣지 않는다.
8. 시연 가능한 기능과 최종 완성 그림을 구분해서 표현한다.

# 12. 완료 기준

1. PPT 목차가 확정된다.
2. 타겟 고객별 메시지가 정리된다.
3. 서비스 최종 그림이 한 장으로 설명된다.
4. 현재 구현현황이 정리된다.
5. 제안서 반영 기능과 제외 기능이 구분된다.
6. 개발 완료 타임라인이 정리된다.
7. 예상 판매가와 목표 판매가가 정리된다.
8. 가격 모델이 1개 이상 제안된다.
9. 고객 테스트 계획이 정리된다.
10. 슬라이드별 핵심 메시지가 작성된다.
11. 발표자 멘트 초안이 작성된다.

# 13. AI에이전트 첨부용 짧은 실행 프롬프트

```markdown
첨부한 MD 파일을 기준으로 HO 인사실장님 인터뷰를 진행해줘.

목표는 인재개발·포스트매칭 서비스의 B2B 제안서 PPT를 만들기 위한 내용을 40분~1시간 안에 파악하는 것이야.

질문은 한 번에 3개 이하로 하고, 타겟 고객, 고객 문제, 서비스 최종 그림, 현재 구현현황, 개발 완료 타임라인, 예상 판매가/목표 판매가, 고객 테스트 계획을 중심으로 물어봐줘.

HTML 파일이나 기존 구현물이 있으면 현재 구현현황을 분석하되, 무조건 보존하지 말고 제안서에 반영할 것과 제외할 것을 실장님에게 확인해줘.

답변은 바로 PPT로 바꿀 수 있게 슬라이드별 메시지와 표/도식 아이디어로 정리해줘.

모르는 내용은 추측하지 말고 `[추후확인]`으로 남겨줘.

최종 산출물은 PPT 제안서 생성용 PRD 형식으로 작성해줘.

이제 오프닝 멘트 후 1단계 질문부터 시작해줘.
```
